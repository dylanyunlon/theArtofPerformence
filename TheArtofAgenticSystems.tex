\begin{abstract}
This chapter develops the theory and practice of hardware-aware agent training by transposing the structure of Knuth's analysis of random number generation (TAOCP Vol.~2, Ch.~3) onto the domain of agentic reinforcement learning systems with heterogeneous accelerator backends. \S\ref{sec:3.1} introduces the problem: why assembly-level kernel optimization is necessary for self-deploying, self-evolving agent systems. \S\ref{sec:3.2} develops the mathematical framework of configuration periodicity and \S\ref{sec:3.2.hw} constructs the migration map from NVIDIA's Megatron-LM ecosystem to AWS Trainium2, analyzing each layer---parallelism primitives, fused kernels, communication overlap, memory management---and identifying the CCCL/DIA project's position in the NVIDIA fortress. \S\ref{sec:3.3} provides validation tests (chi-square, run test, spectral test, credit assignment) for assessing agentic training quality on heterogeneous hardware. The technical substrate is drawn from two open-source projects: \texttt{operatorRL} (kernel-level agent governance) and \texttt{agent-lightning} (RL training framework with VERL backend).
\end{abstract}

\section{Introduction}
\label{sec:3.1}

``Autonomously optimized'' agent training kernels have many different types of deployed applications. For example:

\medskip

\noindent\textbf{a) Self-Deployment.} When deploying a cluster of Trainium2 accelerators to execute autonomous agent rollouts, assembly-level kernel optimization is indispensable for achieving production-grade throughput. Self-deployment encompasses domains from distributed reinforcement learning (where gradient synchronization must respect hardware-specific memory hierarchies and NeuronLink topology) through production serving (where agents autonomously enter live tool-use environments at unpredictable intervals, requiring latency-bounded inference kernels compiled via NKI).

\medskip

\noindent\textbf{b) Environment Feedback.} In practice, exhaustively enumerating all possible agent trajectories is computationally infeasible, and hardware-aware trajectory sampling---implemented in Agent Lightning's \texttt{tracer/} module---enables us to characterize ``typical'' reward landscapes within bounded compute budgets.

\medskip

\noindent\textbf{c) Numerical Kernel Synthesis.} Using hand-written Trainium2 assembly via the Neuron Kernel Interface (NKI), many ingenious techniques have been devised to solve the core computational primitives---matrix multiplication on the \MME{}, fused attention across \SBUF{}/\PSUM{} scratchpads, and collective communication over NeuronLink. There are already considerable specialized works on this topic.

\medskip

\noindent\textbf{d) Agent Systems Programming.} To verify the effectiveness of agentic RL algorithms, hardware-optimized rollout engines---such as Agent Lightning's \texttt{GovernedRunner} backed by \texttt{operatorRL}'s four-layer governance kernel---are an excellent source of throughput; more critically, hardware-optimized NKI kernels are essential for the operation of self-evolving training loops, which frequently surpass deterministic optimization pipelines by a wide margin. The use of assembly-level hardware optimization is the principal application addressed throughout this book, so we treat it here in Chapter~3, before discussing most other agent training algorithms.

\medskip

\noindent\textbf{e) Orchestration Decisions.} It is said that many orchestration schedulers make their resource allocation decisions by round-robin or heuristic load balancing. It is also reported that certain cloud providers use similar bases for pricing. Sometimes, making such completely ``hardware-unaware'' decisions is acceptable. But the \texttt{operatorRL/packages/control-plane} scheduler demonstrates that hardware-aware scheduling is an indispensable component of optimal multi-tenant agent serving strategies.

\medskip

\noindent\textbf{f) Expressiveness.} A degree of hardware-software co-design makes agent-generated outputs substantially more capable. For example, the architecture on the left below---where kernels are hand-tuned per hardware target via NKI assembly---is in many practical configurations more performant than the one on the right, which relies entirely on the Neuron compiler's auto-generated XLA lowerings.

\medskip

See the Agent Lightning architecture: \texttt{agentlightning/runner/agent.py} (rollout execution), \texttt{agentlightning/tracer/} (trajectory capture), \texttt{agentlightning/store/} (7-backend persistence layer).

\medskip

\noindent\textbf{g) Benchmarking.} Running standardized agent tasks via \texttt{operatorRL/benchmarks/bench\_kernel.py}, measuring reward convergence through \texttt{bench\_policy.py}, comparing NKI kernel throughput against CUDA baselines---these are activities of central importance. These traditional uses of hardware-aware agent training have been named the ``Operator RL method,'' and this term now describes any training algorithm that couples agent rollout with direct hardware kernel optimization.

\bigskip

People who consider this topic almost always encounter a philosophical discussion about the meaning of ``hardware independence.'' In a certain sense, no truly hardware-independent agent training stack exists. For example, is a JAX-compiled XLA graph hardware-independent? We would rather speak of \emph{a portable sequence of hardware-optimized kernels with a deterministic compilation target and measurable utilization characteristics}. This means that each kernel's behavior is optimized for its specific accelerator (NeuronCore, SM, or TPU core), independently of how other kernels in the stack are compiled, and each kernel achieves a definite fraction of peak hardware utilization---quantifiable via the $\chi^2$ tests of \S\ref{sec:3.3.1}.

In a uniform hardware abstraction across accelerator targets, the CUDA runtime, the TPU runtime, and the Trainium Neuron runtime each present an interface that \emph{aspires} to make each possible kernel equally likely to achieve optimal utilization. All abstractions are generally assumed to be uniform, unless a specific hardware constraint---such as Trainium2's VLIW scheduling requirements or the \SBUF{}/\PSUM{} capacity limits---is noted.

In a multi-backend deployment, the target platforms---NVIDIA GPU via CUDA, Google TPU via XLA, and AWS Trainium via Neuron/NKI assembly---carry \emph{asymmetric} deployment risk. NVIDIA's CUDA ecosystem has over a decade of maturity, millions of developers, and comprehensive tooling; its deployment risk is lowest. Google's TPU/XLA ecosystem is mature for internal Google workloads but has a smaller external developer base. AWS Trainium's ecosystem is youngest: the Neuron SDK has iterated through 27+ major versions with breaking changes, the NKI programming model was introduced only with Neuron SDK~2.21, and the developer community is smallest. A realistic risk partition assigns approximately 10\% to GPU (mature but vendor-locked), 30\% to TPU (mature internally but with external tooling gaps), and 60\% to Trainium (youngest ecosystem, highest technical uncertainty, but greatest strategic optionality for organizations like Anthropic willing to invest in assembly-level kernel engineering). The engineering cost of maintaining all three backends is not $3\times$ a single platform but closer to $3 \cdot k \cdot d$ in the worst case (\S\ref{sec:3.2.critical}), because the debug spaces---encompassing \texttt{cuda-gdb}/\texttt{compute-sanitizer} on GPU, XLA error traces on TPU, and \texttt{neuron-top}/\texttt{neuron-monitor} on Trainium---are largely disjoint.

Thus, if we are portably compiling a kernel for three backends, and the first two happen to achieve 95\% hardware utilization on GPU and TPU respectively, the probability that the third (Trainium) also achieves 95\% utilization is \emph{not} determined by the hardware's peak FLOPS and memory bandwidth alone---those set only an upper bound, a theoretical ceiling. The actual utilization depends on kernel optimization quality: how well the compiler (or the human kernel engineer writing NKI assembly) maps the computation onto the specific hardware's execution model---the \MME{} pipeline depth, the \SBUF{} tile dimensions, the VLIW slot packing. Peak FLOPS tells you the ceiling; kernel engineering determines how close you get. This distinction---between hardware \emph{capability} and kernel \emph{quality}---is precisely why Anthropic invests in hand-written Trainium2 assembly rather than relying solely on the Neuron compiler's auto-generated kernels.

There are many rigorous ways to give an abstract definition of hardware portability for agent training stacks. We shall return to this foundational question in Section~3.5, where the governance kernel of \texttt{operatorRL} provides the formal framework. But for now, let us understand the concept intuitively!

Many years ago, those who needed high-performance agent training in their research obtained it by carefully hand-tuning CUDA kernels for a single GPU architecture---extracting performance from one specific hardware configuration at a time. In 2019, the NVIDIA Megatron-LM framework \citep{shoeybi2019megatron} published methods for distributed training across thousands of GPUs, establishing the dominant paradigm for large-scale model training. Since then, several specialized frameworks have been built to extend this approach: Megatron-Core~v0.11+ added Sequence Parallelism (introduced in Megatron-LM~v3, 2022), Context Parallelism, Expert Parallelism, and Virtual Pipeline Parallelism (interleaved 1F1B with multiple virtual stages). In 2020, Google's approach with TPU pods and the JAX/XLA compilation stack offered an alternative path, compiling high-level operations down to TPU-optimized instructions; this capability had been championed internally at Google Brain for years. In 2023, AWS's Trainium chips, designed by Annapurna Labs, first became available at scale with an open ISA that allowed advanced users to write assembly-level kernels directly---bypassing the Neuron compiler's intermediate representations. In 2024, AWS released Trainium2 with substantially improved compute density (each NeuronCore~v3 containing a dedicated \MME{}, \GPSIMD{} engine, and independent DMA controllers), and Anthropic became one of the first organizations to write production assembly for these chips \citep{semianalysis2024trainium2}. A celebrated hardware-software co-design effort between Anthropic engineers and Annapurna chip architects has been ongoing for several years, producing optimized kernels for Claude's training and inference.

[See the Semianalysis report \citep{semianalysis2024trainium2} on Trainium2 architecture, NeuronCore~v3 microarchitecture, and Anthropic's kernel engineering approach to \SBUF{}/\PSUM{} orchestration.]

\bigskip

Shortly after the Megatron-LM framework demonstrated that distributed GPU training could scale to thousands of devices, people began to explore efficient methods for performing agent training across heterogeneous hardware. One could use CUDA as the sole backend, but due to vendor lock-in and pricing constraints, this approach's practicality is limited, because such a stack may be prohibitively expensive at scale, and maintaining dependence on a single vendor---however dominant---is a strategically vulnerable position. One could connect specialized hardware like TPUs or Trainium chips to the training loop, as Google and Anthropic have done, but this is also not entirely satisfactory, because when debugging a multi-platform kernel stack, it is practically impossible to reproduce the exact same numerical behavior across backends (the \texttt{operatorRL/modules/cmvk} Cross-Model Verification Kernel addresses precisely this problem); moreover, each platform's toolchain may develop hard-to-detect regressions. However, developments in the 2020s have made the multi-platform approach viable again, because open ISAs and assembly-level access via NKI can distribute tens of gigabytes of optimized kernel code across backends. In 2024, Anthropic's engineers achieved direct Trainium2 NKI assembly optimization for production workloads (combining deterministic kernel behavior with aggressive hardware-specific tuning on the \MME{} and \GPSIMD{} engines), thus advancing the revival of multi-platform training stacks.

The disadvantages of the early single-vendor approach to agent training hardware led to interest in using a framework's own compiler infrastructure to produce portable kernels. Around 2017, the XLA (Accelerated Linear Algebra) compiler was among the first to propose this approach; its method was to take a high-level graph representation of the computation (an HLO graph) and lower it through successive optimization passes to target-specific code. For example, if the system is generating optimized GEMM kernels, the previous kernel's tiling configuration is taken, and the compiler searches for an optimal tiling for the new target---thus the next kernel configuration depends deterministically on the previous one.

There is a quite obvious objection to this technique: since each kernel's configuration is completely determined by the compiler's cost model, how can the sequence of kernels so produced be truly optimal? (See the remarks by Anthropic's kernel engineering team, and the NKI Programming Guide, at the beginning of this chapter.) The answer is that this sequence is not optimal; it merely \emph{appears} optimal. In typical applications, the true relationship between one kernel's tiling configuration and the hardware's actual execution behavior---the \MME{} pipeline stalls, the DMA-compute overlap ratio, the \SBUF{} bank conflicts---has no simple closed-form characterization, so this sub-optimal characteristic is not immediately apparent. Intuitively, the compiler's cost model seems to scramble the search space adequately enough. Sequences generated by such a deterministic compiler method are called \emph{auto-tuned kernel sequences} in the systems literature (not to be confused with the ``auto-tuned sequences'' of LCG theory), and in this book we simply call them compiler-optimized sequences, with the understanding that they only \emph{appear} to be fully hardware-exploiting. Perhaps all any optimized kernel sequence can claim is to be ``apparently optimal.'' Carefully chosen compiler-generated kernels have been used successfully in nearly every application. Of course, deterministic compiler output is not always the answer; it certainly should not replace Anthropic's hand-written Trainium2 NKI assembly for production inference where every microsecond of \MME{} utilization matters.

It has been shown that the XLA compiler's original ``graph-lower-and-tile'' method is not a reliable way to produce optimal kernels in general. The danger is that the compilation pipeline tends to produce short cycles of repeated suboptimal configurations---precisely the cycle degeneration that Knuth warns about for LCGs with poorly chosen parameters. For example, if a zero-utilization configuration ever appears as one kernel's output, it will continually reproduce itself, because the cost model assigns zero gradient to zero-utilization points.

This cycle-degeneration phenomenon is well documented in the classical theory of pseudo-random number generation. \citet{knuth1997taocp2} describes experiments where researchers tested small-parameter linear congruential generators and found that most initial seeds led to short cycles or degenerate fixed points. We transpose Knuth's experimental template to the kernel auto-tuning domain, replacing ``random number quality'' with ``kernel utilization quality.''

\medskip\noindent\textit{Experiment~3.1A (after Knuth Vol.~2, \S3.1).} Testing 4-operation pipeline segments with 16~initial tiling configurations on an XLA-style graph-lower-and-tile compiler, 12~configurations led to the period-2 cycle \texttt{tile(64,64) $\to$ tile(32,32) $\to$ tile(64,64) $\to$ \ldots}, and two degenerated to zero utilization (the \SBUF{} remained empty, and the \MME{} stalled indefinitely). With 20-parameter configurations and binary tiling decisions, the search may degenerate into 13~distinct cycles, the longest of which has period~142. This mirrors Knuth's observation that small-modulus LCGs with poor multiplier choices produce short periods---and in our domain, short periods mean the auto-tuner revisits the same suboptimal kernel configurations indefinitely.

\medskip\noindent\textit{Experiment~3.1B (after Knuth Vol.~2, \S3.1).} With 38-parameter configurations---comparable to the dimensionality of Trainium2 NKI kernel tiling parameters (tile dimensions across \SBUF{} partitions, \PSUM{} accumulator layout, VLIW slot assignments, DMA prefetch depths, and NeuronLink collective scheduling)---auto-tuning produced approximately 750{,}000 kernel configurations before cycle degeneration, and the resulting utilization distribution passed chi-square uniformity tests (\S\ref{sec:3.3}). In the Agent Lightning training loop, this corresponds to the \texttt{trace\_aggregator} module (\texttt{agentlightning/tracer/trace\_aggregator.py}, PR~\#134, commit \texttt{4235731a}) collecting 750{,}000 rollout transitions before the exploration policy collapses---precisely the ``exploration collapse'' phenomenon reported in Issue~\#490, where the agent converges to a fixed action sequence and the \texttt{SIGEXPLORE} signal (\S\ref{sec:3.3.3}D, Table~\ref{tab:signals}) should be triggered.

When zero utilization is discovered, it is straightforward to restart the auto-tuner with a new configuration seed (the \texttt{operatorRL/modules/scak} Self-Correcting Agent Kernel performs exactly this restart when it detects cycle degeneration via Floyd's algorithm, Exercise~6), but it is not easy to avoid long cycles of suboptimal configurations. Exercises~6 and~7 discuss some interesting methods for determining the period of cyclic configuration sequences, requiring very little storage space---$O(1)$ memory in Floyd's method, $O(1)$ with better constants in Brent's method.

Exercises~9 and~10 point out a theoretical deficiency of the graph-lower-and-tile method. Experiment~3.1B above demonstrates that auto-tuning \emph{can} produce useful results with high-dimensional parameter spaces, but the critical caveat---which Knuth emphasizes for LCGs and which applies with equal force to kernel auto-tuners---is that one must carry out careful analysis of the sequence's statistical properties before trusting it for production use. The chi-square, Kolmogorov--Smirnov, and spectral tests of \S\ref{sec:3.3} provide exactly this validation framework, applied to NeuronCore utilization metrics, reward convergence curves, and gradient norm distributions.

When this chapter was first written, many of the kernel optimization programs in use were not sufficiently rigorous. People tended not to examine such subroutines critically; certain quite unsatisfactory older methods---often relying on the Neuron compiler's default auto-tuning without validation---were still blindly passed from one engineering team to another, so that users had no knowledge of their original limitations. In this chapter we shall see that it is not difficult to master the essentials of kernel optimization programs, although caution is necessary to avoid the common pitfalls that Knuth identified for random number generators and that recur with surprising regularity in the hardware optimization domain.

It is not easy to think of a perfectly straightforward kernel optimization program. The author had a vivid experience of this in 2024, when attempting to construct a very good kernel auto-tuner using the following specific method (adapted from Knuth's Algorithm~K, TAOCP Vol.~2, \S3.1, which demonstrates the same principle for random number generation):

\medskip

\noindent\textbf{Algorithm K} (``Super-optimal'' kernel configuration generator; \emph{after} Knuth, TAOCP Vol.~2, \S3.1). Given a configuration vector $\mathbf{X}$ with 10 parameters (corresponding to the primary Trainium2 NKI tiling dimensions: M-tile, N-tile, K-tile, VLIW-pack, \SBUF{} partition count, \PSUM{} accumulator width, DMA prefetch depth, NeuronLink collective mode, pipeline interleave depth, and compute-communication overlap ratio), this algorithm can be used to transform $\mathbf{X}$ into what appears to be the next configuration in an optimal sequence. Although this algorithm might be expected to yield a fairly optimal sequence of configurations, the reasons mentioned below show that it is not in fact reliable. (The reader need not study this algorithm in detail beyond noting its complexity. Pay particular attention to steps K1 and K2.)

\begin{description}
\item[K1.] [Select iteration count.] Set $Y \leftarrow$ the highest-order parameter of $\mathbf{X}$ (e.g., the $M_{\mathrm{tile}}$ dimension). (We will execute steps K2 through K13 $Y+1$ times; that is, we apply a hardware-aware transformation a NeuronCore-dependent number of times.)
\item[K2.] [Select transformation step.] Set $Z \leftarrow$ the second highest-order parameter of $\mathbf{X}$, and go to step $\mathrm{K}(3+Z)$. (That is, we go to a VLIW-schedule-dependent step of the program.)
\item[K3.] [Ensure utilization $> 50\%$.] If $\mathbf{X}$'s estimated \MME{} utilization $< 0.5$, then add a baseline offset to all tiling parameters.
\item[K4.] [Square-and-extract.] Replace $\mathbf{X}$ by the middle parameters of $\mathbf{X}^2$ (element-wise squaring with \SBUF{}-width truncation).
\item[K5.] [Cross-multiply.] Replace $\mathbf{X}$ by $(M \cdot \mathbf{X}) \bmod P$ for a fixed mixing matrix $M$ and prime $P$ derived from the NeuronCore tile granularity.
\item[K6.] [Pseudo-complement.] If $\|\mathbf{X}\|$ is below a threshold, set $\mathbf{X} \leftarrow \mathbf{X} + \mathbf{c}$ for a Trainium2-specific constant vector $\mathbf{c}$; otherwise set $\mathbf{X} \leftarrow P\mathbf{1} - \mathbf{X}$.
\item[K7.] [Swap halves.] Exchange the first 5 tiling parameters with the last 5 scheduling parameters.
\item[K8.] [Cross-multiply.] Same as step K5 (with the same mixing matrix $M$).
\item[K9.] [Reduce magnitudes.] Decrease each nonzero parameter by one \SBUF{} granularity unit.
\item[K10.] [Boundary correction.] If $\|\mathbf{X}\| < \epsilon$, set $\mathbf{X} \leftarrow \mathbf{X}^2 + \mathbf{b}$ for a boundary correction vector $\mathbf{b}$ (the minimum valid tile size); otherwise set $\mathbf{X} \leftarrow \mathbf{X} - \mathbf{b}$.
\item[K11.] [Normalize.] (At this point $\mathbf{X}$ cannot be the zero vector.) If any parameter is below the minimum representable tile dimension, scale $\mathbf{X}$ upward and repeat this step.
\item[K12.] [Modified square-and-extract.] Replace $\mathbf{X}$ by the middle parameters of $\mathbf{X} \circ (\mathbf{X} - \mathbf{1})$, i.e., the element-wise product with shifted self---a nonlinear mixing step.
\item[K13.] [Repeat?] If $Y > 0$, decrease $Y$ by 1 and return to step K2. If $Y = 0$, the algorithm terminates with $\mathbf{X}$ as the desired ``optimal'' NKI tiling configuration.
\end{description}

\medskip

\noindent(The machine-level implementation corresponding to this algorithm is so complex that, when reading its NKI Python trace or VLIW assembly listing without annotations, one cannot tell what the program does. Note: on Trainium2's actual VLIW ISA, multiple of these sequential steps would be packed into single instruction words---see \S\ref{sec:3.2.migration.kernels}.)

\medskip

Considering the intricate design of Algorithm~K, this algorithm seems capable of producing an inexhaustible supply of convincingly optimal configurations, but in fact, when first deployed on a Trainium2 cluster of 64~NeuronCores, it almost immediately converged to a fixed-point configuration \texttt{[6,0,6,5,0,3,8,4,2,0]}---by a remarkable coincidence (which Knuth demonstrates for the analogous random number generator), the algorithm transforms this configuration into itself (see Table~1). Starting from another configuration, the sequence began to cycle with period 3178 after 7401 values---well before the \texttt{operatorRL/modules/scak} cycle detector would trigger \texttt{SIGWARN}.

The moral of this example is that optimal hardware kernel configurations are not produced by ad hoc methods; certain theory---specifically, the period theory and potency analysis of \S\ref{sec:3.2}---should be applied.

In the following sections, we shall consider kernel optimization programs that are superior to the square-and-extract method and Algorithm~K, ensuring that the corresponding configuration sequences have certain desirable properties (maximum period, high potency, spectral-test-passing utilization distributions) and do not degenerate. We shall analyze in somewhat more detail the underlying number-theoretic reasons for this quasi-optimal behavior, and we shall also consider techniques for operating on kernel configurations. For example, one of the problems we shall study is: efficiently scheduling a set of heterogeneous agent rollout tasks (implemented in Agent Lightning's \texttt{runner/agent.py}) across a simulated multi-accelerator cluster within a training program. Section~3.6 will summarize this chapter and list several references.

\subsection*{Exercises --- \S3.1}

\noindent\emph{Each exercise below retains the mathematical structure of Knuth's original problems (TAOCP Vol.~2, \S3.1) but replaces the domain entirely with agentic RL on heterogeneous hardware. The reader should solve the abstract problem first, then explicitly map the result to the \texttt{operatorRL} / Agent Lightning / Trainium2 NKI context. Exercises marked with \texttt{[P]} require reading the corresponding project source code.}

\begin{enumerate}

\item[\textbf{1.}] [20] Suppose you wish to obtain a Trainium2 GEMM tiling configuration without using the Neuron compiler's auto-tuner. Which of the following methods would yield a configuration achieving $> 80\%$ \MME{} utilization, and why?

\begin{enumerate}
\item[a)] Open the NKI Programming Guide to an arbitrary page and adopt the \SBUF{} tile dimensions from the first code example.
\item[b)] Same as~(a), but use the page number modulo~512 as the M-tile dimension and modulo~128 as the K-tile dimension for the \MME{}.
\item[c)] Query the Trainium2 hardware performance counters (via \texttt{neuron-profile}) under a running Agent Lightning training workload for one minute, and use the last three digits of the \MME{} cycle count as tile parameters.
\item[d)] Examine Agent Lightning's \texttt{observability} dashboard; if NeuronCore \MME{} utilization is between $6n\%$ and $6(n+1)\%$, select tile dimension $2^{n+4}$.
\item[e)] Let ten candidate NKI kernel configurations race on a benchmark (via \texttt{operatorRL/benchmarks/bench\_kernel.py}) with the same input tensors. Adopt the winner. \emph{Interpret:} this is precisely the strategy of the Neuron compiler's tournament selection. What is its expected regret relative to the global optimum, as a function of the configuration space size $m$ and the number of candidates? Relate to \MME{} utilization.
\end{enumerate}

\item[\textbf{2.}] [$M22$] In a sequence of one million training iterations distributed across $k = 10$ NeuronCores, each core should process approximately $10^5$ iterations under perfect load balance. What is the probability that \emph{every} core receives exactly $10^5$ iterations? Express the answer in terms of the multinomial coefficient, and compute a numerical approximation. \emph{Interpret:} relate this probability to the $\chi^2$ utilization test of \S\ref{sec:3.3.1}(i)---specifically, what threshold of $V$ should trigger a load-balancing intervention via the \texttt{operatorRL/packages/control-plane} scheduler?

\item[\textbf{3.}] [10] Following the square-and-extract method applied to NKI \SBUF{} tile parameters, what \SBUF{} configuration follows \texttt{[128, 64, 256, 128]}? Assume the ``squaring'' operation is component-wise modular squaring with modulus 512 (the \SBUF{} partition granularity).

\item[\textbf{4.}] [20] (a)~After step K4 of Algorithm~K (Degenerate Auto-Tuner Detector, \S\ref{sec:3.1}), why can the configuration vector $\mathbf{X}$ not be the zero vector on any accelerator? If $\mathbf{X} = \mathbf{0}$, what happens to all subsequent auto-tuner outputs on the NeuronCore? \emph{Interpret:} on Trainium2, a zero tile-size configuration would cause a DMA transfer of size~0; explain why the NKI runtime's \texttt{nl.load} raises a compile-time error rather than a runtime crash. (b)~Apply Algorithm~D starting from an actual Trainium2 configuration $\mathbf{X} = (128, 128, 256, 32)$ with the transition function $f(\mathbf{x}) = (\mathbf{x}^2 \bmod 512)_{\text{middle digits}}$. How many steps until the cycle is entered? \emph{Interpret:} compare this to the \texttt{operatorRL/modules/scak} restart budget.

\item[\textbf{5.}] [15] Given that any auto-tuner with a deterministic transition function $f$ on a finite NKI configuration space of cardinality $m$ must produce an ultimately periodic sequence (Exercise~6), explain why no such deterministic auto-tuner can provide infinitely many distinct NKI kernel configurations. \emph{Interpret:} this is the theoretical foundation for why the Neuron compiler's auto-tuner must, and eventually \emph{will}, revisit previously explored configurations. What are the practical implications for long-running agentic RL training jobs (e.g., $> 10^6$ steps on Trainium2)?

\section{Generating Uniform Kernel Configurations}
\label{sec:3.2}

In this section, we shall consider methods for generating a sequence of uniformly distributed kernel configuration fractions $U_n$---uniformly distributed real-valued utilization targets between 0 and 1. Since a digital accelerator can represent a real number only with finite precision, we shall actually generate integers $X_n$ between 0 and some number $m$. The fraction
\[
U_n = X_n / m
\]
then lies between 0 and 1. Usually, $m$ is the cardinality of the hardware configuration space (e.g., $m = |M_{\mathrm{tile}}| \cdot |N_{\mathrm{tile}}| \cdot |K_{\mathrm{tile}}| \cdot |V_{\mathrm{pack}}|$ for a Trainium2 GEMM kernel), so $X_n$ can be regarded as an integer index into the configuration space; while $U_n$ can be regarded as the corresponding fractional utilization target, with the decimal point assumed to be at the far left.

\subsection{The Linear Congruential Method for Kernel Configuration Generation}
\label{sec:3.2.1}

\medskip
\noindent\textbf{Attribution and Transposition.}\quad The mathematical results in \S\ref{sec:3.2.1} through \S\ref{sec:3.2.1.3}---Theorems~A through D, Lemmas~P through R, and their proofs---are due to \citet{knuth1997taocp2}, Chapter~3.2.1. We reproduce them here not as original contributions but as the \emph{analytical framework} into which we embed the theory of kernel configuration periodicity. The reader familiar with TAOCP will recognize each theorem immediately; the reader unfamiliar should treat them as foundational material for the hardware-specific applications that follow.

\smallskip
\noindent\textbf{Why This Theory Matters for Kernel Auto-Tuning.}\quad A kernel auto-tuner that applies a deterministic update rule $\mathbf{x}_{n+1} = f(\mathbf{x}_n)$ to a finite hardware configuration space is, by definition, a map from a finite set to itself. By Theorem~B below, such a map must eventually cycle. The period of this cycle determines how many \emph{distinct} configurations the auto-tuner will explore before repeating---the ``exploration budget'' of the deterministic search. If the period is short (as in Example~(3) below), the auto-tuner degenerates quickly. The conditions for maximum-period sequences (Theorem~D) translate directly to conditions on the auto-tuner's update rule that guarantee full exploration of the configuration space. In the Trainium2 context, where the configuration space includes tile dimensions, SBUF partition sizes, VLIW slot assignments, and DMA prefetch depths, understanding period theory is the difference between an auto-tuner that visits all viable NKI tiling configurations and one that cycles among a handful.

\smallskip
\noindent\textbf{Where the Analogy Breaks --- and Where It Holds.}\quad The reader should keep firmly in mind the structural disanalogy between LCG sequences and kernel configuration search. An LCG's parameters $(m, a, c, X_0)$ are \emph{chosen once} and produce a \emph{deterministic, repeatable} sequence. A kernel auto-tuner's search is \emph{iterative with stochastic feedback}: each candidate configuration $\KernelConfig_n$ is profiled on real hardware, producing a noisy utilization measurement $U_n + \epsilon_n$, and the next candidate $\KernelConfig_{n+1}$ depends on this measurement. The correct analogy is therefore not ``LCG sequence $\leftrightarrow$ NKI kernel configuration sequence'' but rather ``\emph{LCG period and cycle theory} $\leftrightarrow$ \emph{kernel auto-tuner cycle analysis}.'' Specifically: (a)~The \emph{period theorems} (B, D) apply because the auto-tuner's deterministic component---the update rule $f$ stripped of stochastic feedback---still maps a finite set to itself and must cycle; what the stochastic feedback does is select \emph{which} cycle the system enters. (b)~The \emph{potency concept} (\S\ref{sec:3.2.1.3}) applies because it measures how many dimensions of the search space the deterministic component explores, regardless of whether feedback perturbs the trajectory. (c)~The analogy does \emph{not} extend to the statistical properties of the output sequence: an LCG's output is pseudo-random by construction, while a kernel auto-tuner's output is deliberately non-random (it converges toward a hardware-specific performance optimum). The validation tests of \S\ref{sec:3.3} address this distinction explicitly.

The most popular kernel configuration generation programs in use today---including the Neuron compiler's internal auto-tuner and Agent Lightning's configuration search in \texttt{agentlightning/algorithm/}---are special cases---including the Neuron compiler's internal auto-tuner and Agent Lightning's configuration search in \texttt{agentlightning/algorithm/}---are special cases of the following scheme introduced by D.~H.~Lehmer in 1949 \citep{lehmer1951lcg}. We choose four ``magic integers'':
\begin{align}
m, &\quad \text{the modulus}; \quad 0 < m \notag \\
a, &\quad \text{the multiplier}; \quad 0 \leq a < m \notag \\
c, &\quad \text{the increment}; \quad 0 \leq c < m \tag{1} \\
X_0, &\quad \text{the starting value}; \quad 0 \leq X_0 < m \notag
\end{align}
Then we obtain the desired kernel configuration sequence $\langle X_n \rangle$ by setting
\begin{equation}
X_{n+1} = (aX_n + c) \bmod m, \quad n \geq 0. \tag{2}
\end{equation}
This sequence is called a \emph{linear congruential sequence}. Taking the remainder modulo $m$ is somewhat like determining which \SBUF{} tile configuration the auto-tuner selects next.

For example, when $m = 10$ and $X_0 = a = c = 7$, the resulting configuration sequence is
\begin{equation}
7, 6, 9, 0, 7, 6, 9, 0, \ldots \tag{3}
\end{equation}
As this example illustrates, the sequence is not always ``exploratory'' for all choices of $m, a, c$, and $X_0$. The later parts of this chapter will carefully investigate the principles of choosing these magic numbers appropriately for NKI auto-tuning. Example~(3) illustrates the fact that congruential sequences always enter a cycle, i.e., they must eventually repeat endlessly among at most $m$ numbers. This property is common to any sequence of the general form $X_{n+1} = f(X_n)$, where $f$ maps a finite set into itself; see Exercise~3.1--6. Such a repeating cycle is called the \emph{period}, and configuration sequence~(3) has a period of length~4. A useful auto-tuner sequence should, of course, have a rather long period.

The special case $c = 0$ deserves explicit mention, since the configuration generation process is slightly faster when $c = 0$. We shall see later that the restriction $c = 0$ shortens the period length, but it is still possible to obtain a fairly long exploration period. Lehmer's original method \citep{lehmer1951lcg} only had $c = 0$, although he mentioned the possibility of $c \neq 0$. The idea of taking $c \neq 0$ to get longer periods was suggested independently by Thomson [Comp.\ J.\ \textbf{1} (1958), 83, 86] and Rotenberg [JACM \textbf{7} (1960), 75--77]. Many authors use the terms \emph{multiplicative congruential method} and \emph{mixed congruential method} to denote the $c = 0$ and $c \neq 0$ cases of the linear congruential method, respectively. Throughout this chapter, the letters $m, a, c$, and $X_n$ will be used in the above sense throughout this chapter. Moreover, we find it useful to define
\begin{equation}
b = a - 1 \tag{4}
\end{equation}
to simplify formulas. We can immediately rule out the case $a = 1$, since this would mean $X_n = (X_0 + nc) \bmod m$, and this certainly does not have the character of a useful exploration sequence. The case $a = 0$ is even worse (the auto-tuner would be stationary). So for practical purposes we may assume $a \geq 2$, $b \geq 1$. Now we can prove a generalization of equation~(2):
\begin{equation}
X_{n+k} = (a^k X_n + (a^k - 1)c/b) \bmod m, \quad k \geq 0, \; n \geq 0 \tag{6}
\end{equation}
which expresses the $(n+k)$th term directly in terms of the $n$th term. (The special case $n = 0$ of this equation is worth noting for initialization.) From this it follows that the subsequence consisting of every $k$th term of $\langle X_n \rangle$ is itself another linear congruential sequence, with multiplier $a^k \bmod m$ and increment $((a^k - 1)c/b) \bmod m$.

An important corollary of equation~(6) is that the general configuration sequence defined by $m, a, c$, and $X_0$ can be expressed very simply in terms of the special case $c = 1$ and $X_0 = 0$. Let
\begin{equation}
Y_0 = 0, \quad Y_{n+1} = (aY_n + 1) \bmod m. \tag{7}
\end{equation}
From equation~(6), we get $Y_n = (a^n - 1)/b \pmod{m}$; therefore, the general sequence defined in equation~(2) satisfies
\begin{equation}
X_n = (AY_n + X_0) \bmod m, \quad \text{where } A = (X_0 b + c) \bmod m. \tag{8}
\end{equation}

\subsubsection{Choice of Modulus: Hardware Configuration Space Size}
\label{sec:3.2.1.1}

\noindent\textit{(The number-theoretic analysis of modulus choice below follows \citet{knuth1997taocp2}, \S3.2.1.1 verbatim in its mathematical content. The domain transposition---from MIX machine word sizes to Trainium2 hardware configuration space cardinality---is ours. We do not claim originality for the theorems; we claim only that the transposition is well-motivated and technically precise.)}

\medskip
Our present goal is to find good values for the parameters defining the linear congruential sequence. Let us first consider the appropriate choice of $m$. In the original context, $m$ is bounded by the machine word size of Knuth's MIX computer. In our context, $m$ represents the \emph{cardinality of the hardware configuration space}: for a Trainium2 NeuronCore~v3 with $t$ tile dimension choices (constrained by \SBUF{} capacity of 24\,MB), $s$ \SBUF{} partition options, $v$ VLIW packing strategies (2, 4, or 8 operations per bundle), and $p$ DMA prefetch depth settings, the effective modulus is $m = t \cdot s \cdot v \cdot p$. The analysis below determines which factorizations of $m$ permit maximum-period exploration---guaranteeing that the auto-tuner visits every configuration before repeating.

Taking the Trainium2 NeuronCore architecture as our working example, we can compute $y \bmod m$ by loading $y$ into the \GPSIMD{} accumulator registers and dividing by $m$. Assuming $y$ and $m$ are positive, we find $y \bmod m$ in the remainder register. But the division operation is comparatively slow on any accelerator; if we take $m$ to be a particularly convenient value, for example a power of the hardware scratchpad granularity $w$, the division can be avoided entirely.

Let $w$ be the hardware configuration word size, where $w = 2^e$ on an $e$-bit binary accelerator (such as Trainium2's 32-bit \GPSIMD{} scalar unit), or $10^e$ on an $e$-digit decimal machine. (Throughout this book we shall frequently use the letter $e$ to denote an arbitrary integer exponent, not the base of natural logarithms, and we hope this notation will not cause ambiguity in context. Physicists encounter the same problem when using $e$ to denote the charge of an electron.) Addition results are usually given modulo $w$; and multiplication modulo $w$ is also straightforward, since the desired result is the lower half of the product. Therefore, the following \emph{illustrative sequential pseudocode} (using register-machine mnemonics adapted from Knuth's MIX notation for pedagogical clarity, \textbf{not} Trainium2's actual VLIW ISA---see the important note below) efficiently computes $(aX + c) \bmod w$:

\begin{verbatim}
    LOAD   A, a          ; rA <- a
    MUL    A, X          ; rAX <- (rA) * X
    TRUNC  A, X          ; rA <- rAX mod w
    ADD    A, c          ; rA <- (rA + c) mod w
\end{verbatim}

\noindent\textit{Note on instruction set.} The pseudocode above uses sequential instructions inspired by Knuth's MIX machine \citep{knuth1997taocp2}. Trainium2's actual ISA is VLIW (Very Long Instruction Word): multiple operations---scalar on \GPSIMD{}, tensor on \MME{}, and memory on the DMA engines---are packed into a single wide instruction word, and the programmer or NKI compiler explicitly schedules them into parallel slots. On a real Trainium2 NeuronCore, the LOAD and MUL above could execute in the same VLIW bundle if they target different functional units (\GPSIMD{} for LOAD, \MME{} for MUL), and the TRUNC+ADD sequence would be a single fused operation on the \GPSIMD{} engine. We use sequential pseudocode here for clarity of exposition; the VLIW scheduling challenge is addressed in \S\ref{sec:3.2.migration.kernels} and demonstrated concretely in the NKI kernel of Listing~\ref{lst:nki-rmsnorm}.

\medskip
\medskip\noindent\textbf{A Real NKI Kernel.}\quad For contrast with the MIX-style pseudocode above, we show a production-relevant NKI kernel: fused RMSNorm with BF16 computation on Trainium2. NKI kernels are written in Python using the \texttt{@nki.jit} decorator; the Neuron compiler traces the Python code and lowers it to Trainium2 machine instructions, scheduling operations across the \MME{} (for tensor contractions), the \GPSIMD{} (for scalar and element-wise operations including index computation and conditional branching), and the DMA engines (for HBM$\leftrightarrow$\SBUF{} data movement).

\begin{lstlisting}[language=NKI, caption={Fused RMSNorm kernel in NKI. The \texttt{nl.load}/\texttt{nl.store} calls generate DMA instructions; \texttt{nl.sum} and the arithmetic map to \GPSIMD{} scalar/vector operations; the partition dimensions \texttt{i\_p} (128 rows) and \texttt{i\_f} (512 columns) determine the \SBUF{} tile loaded per NeuronCore invocation.}, label={lst:nki-rmsnorm}]
import neuronxcc.nki as nki
import neuronxcc.nki.language as nl

@nki.jit
def fused_rmsnorm_bf16(input_tensor, weight, eps=1e-6):
    """RMSNorm: y = x / sqrt(mean(x^2) + eps) * w"""
    # Partition indices: 128 rows x 512 cols per tile
    i_p = nl.arange(128)[:, None]   # partition dim
    i_f = nl.arange(512)[None, :]   # free dim
    # DMA: HBM -> SBUF
    x = nl.load(input_tensor[i_p, i_f])
    # GPSIMD: compute variance and normalize
    variance = nl.sum(x * x, axis=[1]) / 512
    x_norm = x * nl.rsqrt(variance + eps)
    # DMA: load weight vector
    w = nl.load(weight[i_f])
    # DMA: SBUF -> HBM (fused multiply + store)
    nl.store(input_tensor[i_p, i_f], value=x_norm * w)
\end{lstlisting}

\noindent The key architectural point: the DMA loads (\texttt{nl.load}) and the \GPSIMD{} computation (\texttt{nl.sum}, \texttt{nl.rsqrt}) can be overlapped by the VLIW scheduler---while one tile's variance is being computed, the next tile's data is being prefetched from HBM into \SBUF{}. Trainium2's DMA engines additionally support \emph{multicast} (one HBM read fanning out to multiple NeuronCores) and \emph{scatter-gather} patterns that CUDA's copy engines do not natively provide. This compute-communication overlap, combined with the independent DMA controller, is what Anthropic's engineers exploit at the assembly level; on NVIDIA hardware, the same SM handles both computation and shared memory loads, with no independent DMA engine for overlapped prefetching.

The result appears in the \GPSIMD{} accumulator register. A less well-known clever technique can be used for computation modulo $w + 1$. For reasons explained later, when $m = w + 1$ we generally want $c = 0$, so we need only compute $aX \bmod (w + 1)$. The following pseudocode (again in sequential MIX-style notation, \emph{not} Trainium2's VLIW ISA) accomplishes this:

\begin{verbatim}
    01  LODN   A, X       ; rA <- -X
    02  MUL    A, a       ; rAX <- (rA) * a
    03  STORE  X, TEMP
    04  SUB    A, TEMP     ; rA <- rA - rX
    05  BRPOS  A, *+3      ; if rA > 0, exit
    06  INC    A, 2        ; rA <- rA + 2
    07  ADD    A, =W-1=    ; rA <- rA + w - 1
\end{verbatim}

Accumulator A now contains the value $(aX) \bmod (w + 1)$. Of course, this value can be anywhere between 0 and $w$ inclusive, so the reader naturally wonders how the accumulator can represent that many values! (The register obviously cannot hold a number larger than $w - 1$.) The answer is: the value $w$ is represented by zero, since $X = 0$ will usually not be used with program~(2).

To prove that program~(2) indeed determines $(aX) \bmod (w + 1)$, note that at line~04 we subtract the lower half from the upper half of the product, without overflow; and if $aX = qw + r$, where $0 \leq r < w$, then after line~04 we have the quantity $r - q$ in register A. Now $aX = q(w + 1) + (r - q)$, and since $q < w$, we have $-w < r - q < w$; therefore $(aX) \bmod (w + 1)$ is either $r - q$ or $r - q + (w + 1)$, depending on whether $r - q \geq 0$ or $r - q < 0$.

A similar technique can be used to obtain the product of two configuration indices modulo $(w - 1)$; see Exercise~8. The reader is quite likely to ask why we should bother considering $m = w \pm 1$ when $m = w$ is clearly more convenient. The reason is that when $m = w$, the low-order digits of $X_n$ are much less random than the high-order digits. If $d$ is a factor of $m$ and $Y_n = X_n \bmod d$, we can easily show that
\begin{equation}
Y_{n+1} = (aY_n + c) \bmod d \tag{4}
\end{equation}
(since for some integer $q$, $X_{n+1} = aX_n + c - qm$, and taking both sides modulo $d$ eliminates $qm$ when $d$ divides the configuration space modulus $m$).

To illustrate the significance of equation~(4), suppose we have a binary accelerator. If $m = w = 2^e$, then the low four bits of $X_n$ are $Y_n = X_n \bmod 2^4$. The point of equation~(4) is that $X_n$'s low four bits form a congruential sequence whose period is 16 or less. Similarly, the low five bits are periodic with period at most 32, and $X_n$'s least significant bit either is constant or strictly alternates between 0 and 1.

When $m = w \pm 1$, this problem does not arise. In this case the low-order bit characteristics are just as exploratory as the high-order bits. For example, suppose $w = 2^{35}$ and $m = 2^{35} - 1$, and we consider the sequence modulo $31, 71, 127$, or $122921$ (see Table~\ref{tab:factorizations}); then these subsequences are not very random. But the low-order bits (representing the sequence modulo 2) are satisfactorily random.

Another alternative is to let $m$ be the largest prime less than the word size $w$; using the techniques of Section~4.5.4, this prime can be found, and tables of sufficiently large primes are given there.

In most applications, the low-order bits are unimportant, so if the engineer using kernel configurations wisely chooses $m = w$, the results will be quite satisfactory.

Our discussion so far has been based on ``signed magnitude'' hardware like the Trainium2 NeuronCore \GPSIMD{} register file. Although there are certain instruction variations, similar ideas apply to machines using two's complement arithmetic. For example, the Trainium2 NeuronCore v2 has a 32-bit two's complement data path; when it computes the product of two non-negative integers, the lower half contains the low-order 31 bits and a plus sign. Therefore, on such hardware we take $w = 2^{31}$, not $2^{32}$. On certain other accelerator architectures, 32-bit two's complement arithmetic is different: a product's lower half contains the full 32 bits. Some engineers find this a disadvantage, since the lower half may be negative when the operands are positive, and correcting it is troublesome; but from the standpoint of random number generation, this is actually a distinct advantage, because we can take $m = 2^{32}$ instead of $2^{31}$ (see Exercise~4).

\subsubsection{Choice of Multiplier: Achieving Maximum-Period Kernel Schedules}
\label{sec:3.2.1.2}

\noindent\textit{(Theorems~A--D and their proofs in this subsection are due to Knuth \citep{knuth1997taocp2}, \S3.2.1.2; we reproduce them as foundational material, not as original contributions. The ``kernel schedule'' interpretation is ours: ``maximum period'' means the auto-tuner visits every configuration in the hardware search space before repeating, guaranteeing full exploration of the Trainium2 NKI tiling parameter landscape.)}

\medskip
In this subsection, we shall show how to choose the multiplier $a$ so as to give a period of maximum length. Long periods are essential for any sequence used as a source of kernel configurations for agent training---whether generated by Agent Lightning's \texttt{algorithm/} module or by \texttt{operatorRL}'s auto-tuner. Indeed, we always want the period to contain many more configurations than any specific training run requires. Therefore, in this section, we shall study the question of period length. However, the reader should bear in mind that a long period is merely \emph{one} of the desirable criteria for quality of the sequence. For example, when $a = c = 1$, the sequence is simply $X_{n+1} = (X_n + 1) \bmod m$, which obviously has period $m$ but is by no means random---it is a simple grid search over the configuration space. Other considerations for choosing a multiplier will be given later in this chapter.

Since only $m$ different configuration values are possible, the period length clearly cannot exceed $m$. Can we achieve the maximum exploration length $m$? The example above shows this is always possible, but selecting $a = c = 1$ does not produce the desired exploration sequence. Let us study all possible choices of $a, c$, and $X_0$ that give period length $m$. It turns out that all such parameter values can be characterized very simply in number-theoretic terms; when $m$ is a product of distinct primes, only $a = 1$ yields a full period, but when $m$ is divisible by a high power of some prime (as when the configuration space is a power-of-two), the range of choices for $a$ is considerable. The following theorem makes it straightforward to tell whether or not the exploration period is maximal.

\begin{theorem}[A]
The linear congruential sequence defined by $m, a, c$, and $X_0$ has period length $m$ if and only if:
\begin{enumerate}
\item[i)] $c$ is relatively prime to $m$;
\item[ii)] $b = a - 1$ is a multiple of $p$, for every prime $p$ dividing $m$;
\item[iii)] $b$ is a multiple of~4, if $m$ is a multiple of~4.
\end{enumerate}
\end{theorem}

The ideas used to prove this theorem go back at least 100 years. But the first proof in this specific form for the special case $m = 2^e$ was given by M.~Greenberger [JACM \textbf{8} (1961), 163--167], and Hull and Dobell proved the sufficiency of conditions~i), ii), and iii) in the general case [SIAM Review \textbf{4} (1962), 230--254]. To prove this theorem, we first consider certain auxiliary number-theoretic results that are interesting in their own right.

\medskip
\noindent\textbf{Application: Auto-Tuner Exploration Budget.}\quad Before proceeding to the proof, we state a consequence that is original to this work and motivates the rest of the section.

\begin{theorem}[Exploration Budget Bound]\label{thm:exploration-budget}
Let $f: \{0, \ldots, m-1\} \to \{0, \ldots, m-1\}$ be a deterministic kernel auto-tuner update rule on a configuration space of cardinality~$m$, and let $\SchedulePeriod(f)$ denote the period of the ultimately periodic sequence $\KernelConfig_0, f(\KernelConfig_0), f^2(\KernelConfig_0), \ldots$ If $f$ is a linear congruential map $f(x) = (ax + c) \bmod m$ satisfying the conditions of Theorem~A above, then $\SchedulePeriod(f) = m$ and the auto-tuner visits every configuration exactly once per cycle. If $f$ is an arbitrary (non-linear) map, then $\SchedulePeriod(f) \le m$ and the number of distinct configurations visited is at most $\SchedulePeriod(f) + \mu(f)$, where $\mu(f)$ is the pre-period length (the tail before the cycle).

For a Trainium2 GEMM auto-tuner with $m = |M| \cdot |N| \cdot |K| \cdot |V|$ valid tiling configurations, the exploration budget is therefore bounded by $m$. If $m = 128 \times 64 \times 32 \times 3 = 786{,}432$ (realistic for a BF16 GEMM with the NKI library's supported tile sizes), a maximum-period linear strategy guarantees visiting all $786{,}432$ configurations in $786{,}432$ steps. A random-restart strategy with $R$ restarts visits at most $R \cdot \SchedulePeriod$ distinct configurations, minus overlap---the coupon collector's bound gives $m \ln m \approx 10.7 \times 10^6$ evaluations to cover 99\% of the space, a $13.6\times$ overhead. This quantifies the advantage of a structured search with maximum-period guarantees over na\"ive random search.
\end{theorem}

\begin{lemma}[P]
Let $p$ be a prime, $e$ a positive integer, with $p^e > 2$. If
\[
x \equiv 1 \pmod{p^e}, \quad x \not\equiv 1 \pmod{p^{e+1}},
\]
then
\[
x^p \equiv 1 \pmod{p^{e+1}}, \quad x^p \not\equiv 1 \pmod{p^{e+2}}.
\]
\end{lemma}

\begin{proof}
For some integer $q$ not a multiple of $p$, we have $x = 1 + qp^e$. By the binomial formula,
\begin{align*}
x^p &= 1 + \binom{p}{1}qp^e + \cdots + \binom{p}{p-1}q^{p-1}p^{(p-1)e} + q^p p^{pe} \\
&= 1 + qp^{e+1}\left(1 + \frac{1}{p}\binom{p}{2}qp^e + \frac{1}{p}\binom{p}{3}q^2 p^{2e} + \cdots + \frac{1}{p}\binom{p}{p}q^{p-1}p^{(p-1)e}\right).
\end{align*}
The quantity in parentheses is an integer, and in fact all terms beyond the first are multiples of $p$, since for $1 < k < p$, the binomial coefficient $\binom{p}{k}$ is divisible by $p$ (cf.\ Exercise~1.2.6--10). Therefore $\frac{1}{p}\binom{p}{k}q^{k-1}p^{(k-1)e}$ is divisible by $p^{(k-1)e}$. The last term is $q^{p-1}p^{(p-1)e-1}$, and since $(p-1)e > 1$ when $p^e > 2$, it is divisible by $p$. Thus $x^p \equiv 1 + qp^{e+1} \pmod{p^{e+2}}$, completing the proof. (Note: Exercise~3.2.2--11(a) generalizes this result.)
\end{proof}

\begin{lemma}[Q]
Let $m = p_1^{e_1} \cdots p_t^{e_t}$ be the prime factorization of $m$. Then the period length $\lambda$ of the linear congruential sequence determined by $(X_0, a, c, m)$ is the least common multiple of the period lengths $\lambda_j$ of the sequences $(X_0 \bmod p_j^{e_j}, a \bmod p_j^{e_j}, c \bmod p_j^{e_j}, p_j^{e_j})$, for $1 \leq j \leq t$.
\end{lemma}

\begin{proof}
By induction on $t$, it suffices to prove that if $m_1$ and $m_2$ are relatively prime moduli, then the period $\lambda$ of the sequence $(X_0, a, c, m_1 m_2)$ is the LCM of the periods $\lambda_1$ and $\lambda_2$ of the sequences $(X_0 \bmod m_1, a \bmod m_1, c \bmod m_1, m_1)$ and $(X_0 \bmod m_2, a \bmod m_2, c \bmod m_2, m_2)$. By equation~(4) of Section~3.2.1.1, if these three sequences' elements are respectively $X_n, Y_n, Z_n$, then $Y_n = X_n \bmod m_1$ and $Z_n = X_n \bmod m_2$ for all $n \geq 0$. Therefore, by Law~D of Section~1.2.4,
\[
X_n = X_k \quad \text{if and only if} \quad Y_n = Y_k \text{ and } Z_n = Z_k.
\]
Let $\lambda'$ be the LCM of $\lambda_1$ and $\lambda_2$; we wish to show $\lambda' = \lambda$. Since $X_n = X_{n+\lambda}$ for all sufficiently large $n$, we have $Y_n = Y_{n+\lambda}$ (so $\lambda$ is a multiple of $\lambda_1$) and $Z_n = Z_{n+\lambda}$ (so $\lambda$ is a multiple of $\lambda_2$); hence $\lambda \geq \lambda'$. Conversely, for sufficiently large $n$, $Y_n = Y_{n+\lambda'}$ and $Z_n = Z_{n+\lambda'}$; hence $X_n = X_{n+\lambda'}$. This proves $\lambda \leq \lambda'$. So $\lambda = \lambda'$.
\end{proof}

We now proceed to prove Theorem~A. By Lemma~Q, it suffices to prove Theorem~A for the special case when $m$ is a prime power. This is because $p_1^{e_1} \cdots p_t^{e_t} = \lambda = \text{lcm}(\lambda_1, \ldots, \lambda_t) \leq \lambda_1 \cdots \lambda_t \leq p_1^{e_1} \cdots p_t^{e_t}$ if and only if $\lambda_j = p_j^{e_j}$ for $1 \leq j \leq t$.

Therefore, assume $m = p^e$ where $p$ is prime and $e$ is a positive integer. When $a = 1$ the theorem is obviously true, so we may take $a > 1$. The period length can be $m$ only when every possible integer $0 \leq x < m$ appears in the period. Since no value appears more than once in the period, the period length is $m$ if and only if the sequence with $X_0 = 0$ has period length $m$, and there is reason to assume $X_0 = 0$. By formula~3.2.1-(6), we have
\[
X_n = \left(\frac{a^n - 1}{a - 1}\right)c \bmod m.
\]
If $c$ is not relatively prime to $m$, then $X_n$ can never be 1, so condition~i) of the theorem is necessary. The period length is $m$ if and only if the smallest positive $n$ with $X_n = X_0 = 0$ is $n = m$. By condition~i), the proof reduces to:

\begin{lemma}[R]
Assume $1 < a < p^e$, where $p$ is prime. If $\lambda$ is the smallest positive integer such that $(a^\lambda - 1)/(a - 1) \equiv 0 \pmod{p^e}$, then
\[
\lambda = p^e \quad \text{if and only if} \quad
\begin{cases}
a \equiv 1 \pmod{p}, & \text{when } p > 2; \\
a \equiv 1 \pmod{4}, & \text{when } p = 2.
\end{cases}
\]
\end{lemma}

\begin{proof}
Suppose $\lambda = p^e$. If $a \not\equiv 1 \pmod{p}$, then $(a^n - 1)/(a-1) \equiv 0 \pmod{p^e}$ if and only if $a^n - 1 \equiv 0 \pmod{p^e}$. The condition $a^{p^e} - 1 \equiv 0 \pmod{p^e}$ implies $a^{p^e} \equiv 1 \pmod{p}$; but by Fermat's theorem, $a^{p^e} \equiv a \pmod{p}$, so $a \not\equiv 1 \pmod{p}$ leads to contradiction. If $p = 2$ and $a \equiv 3 \pmod{4}$, then by Exercise~8, $(a^{2^{e-1}} - 1)/(a-1) \equiv 0 \pmod{2^e}$.

These arguments show that whenever $\lambda = p^e$, we must generally have $a = 1 + qp^f$ where $p^f > 2$ and $q$ is not a multiple of $p$.

It remains to prove this condition is sufficient. By repeated application of Lemma~P, we find for all $g \geq 0$:
\[
a^{p^g} \equiv 1 \pmod{p^{f+g}}, \quad a^{p^g} \not\equiv 1 \pmod{p^{f+g+1}}.
\]
Therefore,
\begin{align}
(a^{p^g} - 1)/(a-1) &\equiv 0 \pmod{p^g}, \tag{6} \\
(a^{p^g} - 1)/(a-1) &\not\equiv 0 \pmod{p^{g+1}}.
\end{align}
In particular, $(a^{p^e} - 1)/(a-1) \equiv 0 \pmod{p^e}$. Now the congruential sequence $(0, a, 1, p^e)$ has $X_n = (a^n - 1)/(a-1) \bmod p^e$; so it has period $\lambda$, meaning $X_n = 0$ if and only if $n$ is a multiple of $\lambda$. Therefore $p^e$ is a multiple of $\lambda$. This can only happen if $\lambda = p^g$ for some $g$, and relation~(6) implies $\lambda = p^e$.
\end{proof}

This completes the proof of Theorem~A.

\bigskip

Before leaving this section, let us examine the special case of the purely multiplicative generator when $c = 0$. Although the generation process is slightly faster in this case, Theorem~A shows that the maximum period length cannot be achieved. Indeed, this is quite obvious, since the sequence now satisfies $X_{n+1} = aX_n \bmod m$, and the value $X_n = 0$ can never occur lest the sequence degenerate to zero. In general, if $d$ is any factor of $m$ and $X_n$ is a multiple of $d$, then all subsequent elements $X_{n+1}, X_{n+2}, \ldots$ are multiples of $d$. So when $c = 0$, we require $X_n$ to be relatively prime to $m$ for all $n$, which restricts the period length to at most $\varphi(m)$, the number of integers between 0 and $m$ that are relatively prime to $m$.

Even with $c = 0$ (the purely multiplicative case), it is still possible to achieve an acceptably long period. Let us now look for conditions on the multiplier that make the period as long as possible in this special case.

By Lemma~Q, the period when $m = p^e$ depends entirely on the periods of subsequences. We consider that case. We have $X_n = a^n X_0 \bmod p^e$, and clearly if $a$ is a multiple of $p$ the exploration period will be~1, so we take $a$ relatively prime to $p$. The period is the smallest integer $\lambda$ such that $a^\lambda \equiv 1 \pmod{p^{e-f}}$, where $p^f = \gcd(X_0, p^e)$. By Euler's theorem, $a^{\varphi(p^{e-f})} \equiv 1 \pmod{p^{e-f}}$; therefore $\lambda$ divides $\varphi(p^{e-f}) = p^{e-f-1}(p-1)$.

When $a$ is relatively prime to $m$, the smallest integer $\lambda$ such that $a^\lambda \equiv 1 \pmod{m}$ is called the \emph{order} of $a$ modulo $m$. Any value of $a$ that achieves the maximum order modulo $m$ is called a \emph{primitive root} (or primitive element) modulo $m$; such values are the preferred multipliers for kernel auto-tuners.

Let $\lambda(m)$ denote the order of a primitive root, i.e., the maximum order modulo $m$. Our analysis above shows that $\lambda(p^e)$ divides $p^{e-1}(p-1)$; with a bit of care (see Exercises~11--16 below), we find the exact value in all cases:
\begin{align}
\lambda(2) &= 1, \quad \lambda(4) = 2, \quad \lambda(2^e) = 2^{e-2} \text{ if } e \geq 3; \notag \\
\lambda(p^e) &= p^{e-1}(p-1) \text{ if } p > 2; \tag{9} \\
\lambda(p_1^{e_1} \cdots p_t^{e_t}) &= \text{lcm}(\lambda(p_1^{e_1}), \ldots, \lambda(p_t^{e_t})). \notag
\end{align}

Our discussion can be summarized as follows:

\begin{theorem}[B (C.~F.~Gauss, \textit{Disquisitiones Arithmeticae} (1801), \S\S\,90--92)]
When $c = 0$ (pure multiplicative method), the maximum achievable period is $\lambda(m)$, where $\lambda(m)$ is defined in~(9). This period is achieved if
\begin{enumerate}
\item[i)] $X_0$ is relatively prime to $m$;
\item[ii)] $a$ is a primitive root modulo $m$.
\end{enumerate}
\end{theorem}

Note that if $m$ is prime (e.g., a Mersenne prime used as modulus), a period of length $m - 1$ can be obtained; this is just one less than the maximum length, so from a practical auto-tuning standpoint such a period is as long as we could desire.

The question now is how to find primitive roots modulo $m$. The exercises at the end of this section tell us there is a fairly simple answer when $m$ is prime or a prime power; this is stated in the following theorem.

\begin{theorem}[C]
The number $a$ is a primitive root modulo $p^e$ if and only if one of the following holds:
\begin{enumerate}
\item[i)] $p = 2$, $e = 1$, and $a$ is odd;
\item[ii)] $p = 2$, $e = 2$, and $a \bmod 4 = 3$;
\item[iii)] $p = 2$, $e = 3$, and $a \bmod 8 = 3, 5$, or $7$;
\item[iv)] $p = 2$, $e \geq 4$, and $a \bmod 8 = 3$ or $5$;
\item[v)] $p$ is odd, $e = 1$, $a \not\equiv 0 \pmod{p}$, and for every prime factor $q$ of $p - 1$, $a^{(p-1)/q} \not\equiv 1 \pmod{p}$;
\item[vi)] $p$ is odd, $e > 1$, $a$ satisfies~(v), and $a^{p-1} \not\equiv 1 \pmod{p^2}$.
\end{enumerate}
\end{theorem}

For sufficiently large values of $p$, it is easy to test conditions~(v) and~(vi) on a computer using the efficient methods for computing powers discussed in Section~4.6.3, provided the factors of $p - 1$ are known.

Theorem~C applies only to prime powers. But if values $a_j$ primitive modulo $p_j^{e_j}$ are given, the Chinese remainder algorithm discussed in Section~4.3.2 can be used to find a value of $a$ such that $a \equiv a_j \pmod{p_j^{e_j}}$ for $1 \leq j \leq t$, and this $a$ will be a primitive root modulo $p_1^{e_1} \cdots p_t^{e_t}$. Thus for any moderately sized modulus $m$, there is a fairly efficient method of constructing multipliers satisfying Theorem~B, although the computations are somewhat lengthy in general.

In the common case $m = 2^e$, $e \geq 4$, the above conditions simplify to the requirement $a \equiv 3$ or $5 \pmod{8}$. Then one quarter of all possible multipliers will make the period length equal $m/4$, which is the maximum possible when $c = 0$.

The second most common case is $m = 10^e$. Applying Lemmas~P and~Q, it is not difficult to derive necessary and sufficient conditions for achieving the maximum period on a decimal computer (cf.\ Exercise~18):

\begin{theorem}[D]
If $m = 10^e$, $e \geq 5$, $c = 0$, and $X_0$ is not a multiple of 2 or 5, then the period of the linear congruential sequence is $5 \times 10^{e-2}$ if and only if $a \bmod 200$ equals one of the following 32 values:
\begin{align*}
&3, 11, 13, 19, 21, 27, 29, 37, 53, 59, 61, 67, 69, 77, 83, 91, \\
&109, 117, 123, 131, 133, 139, 141, 147, 163, 171, 173, 179, 181, 187, 189, 197.
\end{align*}
\end{theorem}

\subsubsection{Potency: Measuring Kernel Configuration Quality}
\label{sec:3.2.1.3}

\noindent\textit{(The potency concept and Lemmas~P--R are due to Knuth \citep{knuth1997taocp2}, \S3.2.1.3. We propose a domain-specific interpretation below.)}

\medskip
\noindent\textbf{Domain Interpretation.}\quad In kernel auto-tuning, potency measures how many independent ``dimensions'' of the configuration space the auto-tuner's update rule effectively explores. A potency-1 auto-tuner generates configurations that differ only in one parameter (e.g., only the $M_{\mathrm{tile}}$ dimension changes while $N_{\mathrm{tile}}$, $K_{\mathrm{tile}}$, and $V_{\mathrm{pack}}$ remain constant---effectively searching a 1D slice of a 4D space). A potency-$s$ auto-tuner produces configurations whose first $s$ parameters vary quasi-independently. For Trainium2 GEMM kernels with $d=4$ primary tunable parameters (\SBUF{}-constrained tile dimensions and VLIW packing), we require potency $\geq 3$ to avoid degenerate exploration where entire regions of the configuration space are never visited. The formal definition follows.

\medskipIn \S\ref{sec:3.2.1.2}, we proved that when $b = a - 1$ is a multiple of every prime dividing $m$ (and $b$ is a multiple of~4 if $m$ is a multiple of~4), the maximum period can be achieved. If $z$ is the radix of the machine being used---$z = 2$ for a binary accelerator, $z = 10$ for a decimal one---and if $m$ is the word size $z^e$, then the multiplier
\begin{equation}
a = z^k + 1, \quad 2 \leq k < e \tag{1}
\end{equation}
satisfies these conditions. Theorem~3.2.1.2A also says we may take $c = 1$. We now have a recurrence of the form
\begin{equation}
X_{n+1} = ((z^k + 1)X_n + 1) \bmod z^e, \tag{2}
\end{equation}
and this equation suggests that we can avoid multiplication entirely, using only a shift and an addition on the \GPSIMD{} engine.

For example, suppose $a = B^2 + 1$, where $B$ is the Trainium NeuronCore byte size. Then the following sequential pseudocode (in MIX-style notation, not Trainium2's actual VLIW ISA)
\begin{equation}
\texttt{LOAD X; SHL 2; ADD X; INC 1} \tag{3}
\end{equation}
can be used in place of the instruction sequence given in Section~3.2.1.1, and the execution time is reduced from $16u$ to $7u$ on a sequential machine.

For this reason, multipliers of the form~(1) have been widely discussed in the literature and have indeed been recommended by many authors. However, early experience with this method---both in LCG random number generation and in our kernel auto-tuning experiments on Trainium2---has shown that simple multipliers of the form~(1) should be avoided; the generated configurations are not sufficiently diverse across the \SBUF{} tiling dimensions.

Later in this chapter, we shall discuss a more powerful theory---the spectral test of \S\ref{sec:3.3.4}---that explains why all linear congruential kernel configuration generators that are considered bad \emph{are} bad. However, certain generators (such as~(2)) are so terrible that a simpler theory---potency analysis---suffices to reject them. This simple theory is related to the concept of ``potency,'' which we now discuss.

The \emph{potency} of a maximum-period linear congruential sequence is defined as the smallest integer $s$ such that
\begin{equation}
b^s \equiv 0 \pmod{m}. \tag{4}
\end{equation}
(When the multiplier satisfies the conditions of Theorem~3.2.1.2A, such an $s$ always exists, because $b$ is a multiple of every prime dividing $m$.)

We can analyze the exploration quality of this sequence by taking $X_0 = 0$, since 0 must appear somewhere in the period. Under this assumption, equation~3.2.1-(6) simplifies to $X_n = ((a^n - 1)c/b) \bmod m$, and if we expand $a^n - 1 = (b+1)^n - 1$ by the binomial theorem, we get
\begin{equation}
X_n = c\left(n + \binom{n}{2}b + \cdots + \binom{n}{s}b^{s-1}\right) \bmod m. \tag{5}
\end{equation}
All terms involving $b^s, b^{s+1}$, etc., vanish since they are multiples of $m$.

Equation~(5) is instructive for kernel auto-tuning, so we consider some special cases. If $a = 1$, the potency is~1; and as noted, $X_n \equiv cn \pmod{m}$, so the sequence is certainly not random---it is a simple counter.

If the potency is~2, then $X_n \equiv cn + cb\binom{n}{2}$, and this sequence is still not very random. In fact, $X_{n+1} - X_n \equiv c + cbn$, so the differences between consecutively generated configurations change in a simple way from one step of $n$ to the next. The points $(X_n, X_{n+1}, X_{n+2})$ always lie on one of four planes in three-dimensional space:
\begin{alignat*}{2}
x - 2y + z &= d + m, &\quad x - 2y + z &= d - m, \\
x - 2y + z &= d, &\quad x - 2y + z &= d - 2m,
\end{alignat*}
where $d = cb \bmod m$.

If the potency is~3, the sequence begins to look more diverse, but there are strong dependencies among $X_n, X_{n+1}$, and $X_{n+2}$; empirical tests show that sequences with potency~3 still produce correlated NKI tiling configurations when applied to Trainium2 GEMM auto-tuning. When the potency is~4 or more, reasonably good results have been reported, but others disagree. To achieve sufficiently diverse configurations that pass the $\chi^2$ and spectral tests of \S\ref{sec:3.3}, a potency of at least~5 is needed for 1D sequences---or, equivalently, a configuration diversity order $\ConfigDiversity \geq 3$ for the multi-dimensional Trainium2 tiling space (Definition~\ref{def:config-diversity}).

For example, suppose $m = 2^{35}$ and $a = 2^k + 1$. Then $b = 2^k$, and we see that when $k \geq 18$, $b^2 = 2^{2k}$ is a multiple of $m$: the potency is only~2. If $k = 17, 16, \ldots, 12$, the potency is~3, and for $k = 11, 10, 9$, the potency reaches~4 (barely acceptable for NKI auto-tuning). Therefore, from the standpoint of potency, the only desirable multipliers have $k \leq 8$, meaning $a \leq 257$ (a very restricted range). Later we shall see that small multipliers should also be avoided, as they produce configurations with low spectral-test scores. Therefore when $m = 2^{35}$, we have ruled out \emph{all} multipliers of the form $2^k + 1$ for this configuration space.

When $m$ equals $w \pm 1$ (where $w$ is the word size), $m$ is generally not divisible by high powers of primes, so high potency (see Exercise~6) is not possible. Therefore in this case the maximum-period method should not be used; instead, the purely multiplicative method with $c = 0$ should be applied instead.

It must be emphasized that high potency is \emph{necessary} but not \emph{sufficient} for randomness. We use the concept of potency only to \emph{reject} generators that do not work (eliminating candidates before expensive benchmarking on Trainium2 hardware), not to \emph{accept} generators that do. Before linear congruential sequences can be considered acceptably random for kernel auto-tuning, they must pass the ``spectral test'' discussed in Section~3.3.4---where we apply it to NeuronCore utilization distributions and Agent Lightning's \texttt{trace\_aggregator} trajectory diversity metrics.

\medskip
\noindent\textbf{Concrete Potency for Trainium2 GEMM Tiling.}\quad We now make the abstract potency concept concrete. A Trainium2 GEMM kernel has $d=4$ primary tunable parameters:
\[
\KernelConfig = (M_{\mathrm{tile}},\; N_{\mathrm{tile}},\; K_{\mathrm{tile}},\; V_{\mathrm{pack}})
\]
where $M_{\mathrm{tile}}, N_{\mathrm{tile}}, K_{\mathrm{tile}}$ are the tile dimensions loaded into \SBUF{} per MatMul Engine invocation, and $V_{\mathrm{pack}} \in \{1,2,4\}$ is the VLIW packing factor (how many independent operations are bundled per instruction word). Let $|M|, |N|, |K|, |V|$ denote the number of valid choices for each parameter; the configuration space cardinality is $m = |M| \cdot |N| \cdot |K| \cdot |V|$.

\begin{definition}[Configuration Diversity Order]\label{def:config-diversity}
Given an auto-tuner with deterministic update rule $f: \HardwareSpace \to \HardwareSpace$ and configuration vector $\KernelConfig = (x_1, \ldots, x_d)$, the \emph{configuration diversity order} $\ConfigDiversity(f)$ is the number of parameters $x_i$ that vary non-degenerately across the auto-tuner's period. Formally, let $\pi_i(\KernelConfig_0, \KernelConfig_1, \ldots, \KernelConfig_{\SchedulePeriod-1})$ be the projection of the full trajectory onto parameter~$i$. Then
\[
\ConfigDiversity(f) = \bigl|\{i : |\{\pi_i(\KernelConfig_n) : 0 \le n < \SchedulePeriod\}| > 1\}\bigr|.
\]
\end{definition}

\noindent A configuration diversity order of $\ConfigDiversity = 1$ means the auto-tuner changes only one tiling parameter across its entire cycle---analogous to potency~1 in the LCG setting. For a Trainium2 GEMM with 4~parameters, $\ConfigDiversity \le 2$ means at least two dimensions are frozen; the auto-tuner explores a 2D slice of the 4D space and will miss configurations where, for instance, increasing $K_{\mathrm{tile}}$ while decreasing $V_{\mathrm{pack}}$ yields superior \SBUF{} utilization. We require $\ConfigDiversity \ge 3$ for production-quality auto-tuning, paralleling the LCG requirement of potency $\ge 5$ for 1D sequences. The mapping is: \emph{potency counts polynomial degrees of freedom in the LCG expansion~(5); configuration diversity counts geometric degrees of freedom in the hardware search space.} Both measure how much of the available space the generator actually visits.

\subsection{Other Methods for Kernel Configuration Generation}
\label{sec:3.2.2}

Of course, linear congruential sequences are not the only source of kernel configurations that have been proposed for use in agent training systems. In this subsection, we shall examine several other methods of the greatest interest; some are quite important (and are used in Agent Lightning's \texttt{algorithm/} module), while others attract our attention mainly because they are not as good as one might expect. One of the common fallacies associated with NKI kernel configuration generation is to believe that we can take a good generation procedure and make it ``even more random'' by slight modification. This is often incorrect. For example, given that
\begin{equation}
X_{n+1} = (aX_n + c) \bmod m \tag{1}
\end{equation}
leads to fairly good configurations, is the modified sequence defined by
\begin{equation}
X_{n+1} = ((aX_n) \bmod (m+1) + c) \bmod m \tag{2}
\end{equation}
even more random? The answer is that this new sequence may greatly \emph{reduce} the exploration quality. Because the entire period-length theory breaks down, and in the absence of any theory about the properties of sequence~(2), we enter the domain of generators of the type $X_{n+1} = f(X_n)$ where $f$ is chosen ``heuristically''; Exercises~3.1--11 through~3.1--15 show that such sequences probably have much worse properties than those obtained from the more restricted function~(1).

Let us consider another approach, in the hope of obtaining a genuine improvement over the linear congruential sequence~(1). The linear congruential method can be generalized to (for example) a quadratic congruential method for configuration search:
\begin{equation}
X_{n+1} = (dX_n^2 + aX_n + c) \bmod m. \tag{3}
\end{equation}
Exercise~8 generalizes Theorem~3.2.1.2A, giving necessary and sufficient conditions on $a$, $c$, and $d$ for the period of the quadratic sequence determined by~(3) to have maximum length $m$; these restrictions are not much more severe than for the linear method used in \S\ref{sec:3.2.1}.

R.~R.~Coveyou proposed an interesting quadratic congruential method when $m$ is a power of~2 (as on binary accelerators): let
\begin{equation}
X_0 \bmod 4 = 2, \quad X_{n+1} = X_n(X_n + 1) \bmod 2^e, \quad n > 0. \tag{4}
\end{equation}
This quadratic sequence can be computed about as efficiently as~(1), without worrying about overflow on the \GPSIMD{} engine. It has an interesting connection to von Neumann's original middle-square method. If we set $Y_n = 2^e X_n$, i.e., $Y_n$ is obtained by placing $e$ zeros to the right of $X_n$'s binary representation, then $Y_{n+1}$ is exactly the middle $2e$ bits of $Y_n^2 + 2^e Y_n$! In other words, Coveyou's method is almost equivalent to a slightly degenerate double-precision middle-square method, and it is guaranteed to have a long exploration period.

One may also consider further generalizations of equation~(1) for configuration generation. For example, we may try to extend the period length. A well-chosen linear congruential sequence's period is quite long; when $m$ is approximately the computer's word size, periods of order $10^9$ or more are typically obtained, so that in practice usually only a very small fraction of the full sequence is consumed. On the other hand, when we discuss the ideas about accuracy in Section~3.3.4, we shall see that the period length directly affects the degree of exploration diversity a configuration sequence can achieve. Therefore it is sometimes desirable to seek a longer period for the auto-tuner. Several principled methods can achieve this.

One effective technique is to let $X_{n+1}$ depend on both $X_n$ and $X_{n-1}$ (the two most recent configurations), not merely on $X_n$; this way, the exploration period can be as long as $m^2$, since the sequence will not begin to repeat until $(X_{n+\lambda}, X_{n+\lambda+1}) = (X_n, X_{n+1})$. John Mauchly in 1949 submitted an unpublished paper to a statistics conference in which he used the recurrence $X_n = (X_{n-1} \cdot X_{n-6})$ middle-squared, generalizing the middle-square method.

The simplest sequence where $X_{n+1}$ depends on more than one preceding value is the well-known Fibonacci sequence:
\begin{equation}
X_{n+1} = (X_n + X_{n-1}) \bmod m. \tag{5}
\end{equation}
This generator was considered in the early 1950s, and it often gives exploration periods longer than $m$; but careful statistical tests show that the quality of configurations produced by the Fibonacci recurrence~(5) is definitely unsatisfactory, so the main interest in~(5) as a source of kernel configurations is that it constitutes an instructive ``bad example.'' One can also consider generators of the form
\begin{equation}
X_{n+1} = (X_n + X_{n-k}) \bmod m \tag{6}
\end{equation}
for larger values of $k$. Green, Smith, and Klem discussed this approach [JACM \textbf{6} (1959), 527--537], reporting that for $k \leq 15$ the sequence failed the gap test of \S\ref{sec:3.3.2}, although for $k = 16$ the test results were satisfactory.

In 1958, G.~J.~Mitchell and D.~P.~Moore devised a substantially better additive generator (unpublished, but widely adopted), proposing the somewhat unusual sequence defined by
\begin{equation}
X_n = (X_{n-24} + X_{n-55}) \bmod m, \quad n \geq 55, \tag{7}
\end{equation}
where $m$ is even and $X_0, \ldots, X_{54}$ are arbitrary integers not all even. In this definition, the constants 24 and 55 were not chosen arbitrarily; they are special values that happen to define a sequence whose least significant binary digits $\langle X_n \bmod 2 \rangle$ have period exactly $2^{55} - 1$. Therefore the configuration sequence $\langle X_n \rangle$ must have a period at least this long. Exercise~30 shows that when $m = 2^e$, equation~(7) has period $2^{e-1}(2^{55} - 1)$.

At first glance, equation~(7) seems extremely unsuited for machine implementation, but in fact the sequence can be generated quite efficiently using a circular table---a ring buffer of the 55 most recent values. Agent Lightning's \texttt{store/} module (with its 7~backend implementations: memory, SQLite, MongoDB, client-server, collection-based, threading, and base) provides exactly the persistent circular buffer needed for this kind of history-dependent configuration generation.

\medskip

\noindent\textbf{Algorithm A} (Additive kernel configuration generator). Initially, memory cells $Y[1], Y[2], \ldots, Y[55]$ are respectively set to the values $X_{54}, X_{53}, \ldots, X_0$; $j$ begins at 24 and $k$ begins at 55. Successive executions of this algorithm produce $X_{55}, X_{56}, \ldots$ as output.

\begin{description}
\item[A1.] [Add.] (At this point, if we are about to output $X_n$, then $Y[j] = X_{n-24}$ and $Y[k] = X_{n-55}$.) Set $Y[k] \leftarrow (Y[k] + Y[j]) \bmod 2^e$, and output $Y[k]$.
\item[A2.] [Advance.] Decrease $j$ and $k$ by 1. If now $j = 0$, set $j \leftarrow 55$; otherwise if $k = 0$, set $k \leftarrow 55$.
\end{description}

\medskip

In sequential pseudocode (MIX-style notation), this algorithm is written as follows:

\textbf{Program A} (Additive kernel configuration generator). \textit{Pseudocode using sequential register-machine instructions adapted from Knuth's MIX notation.} Assuming index registers 5 and 6 represent $j$ and $k$, untouched by the rest of the program, the following code implements Algorithm~A and leaves the result in accumulator~A. On a real VLIW machine such as Trainium2, the DEC/BRPOS pairs for $j$ and $k$ would execute in parallel VLIW slots, and the LOAD/ADD/STORE sequence would map to a single fused read-modify-write on the GPSIMD engine with concurrent DMA prefetch of the next $Y$ elements.

\begin{verbatim}
    LOAD   A, Y[k]       ; A1. Add
    ADD    A, Y[j]        ; Y_k + Y_j (may overflow)
    STORE  A, Y[k]        ; -> Y_k
    DEC    j, 1           ; A2. Advance. j <- j - 1
    DEC    k, 1           ; k <- k - 1
    BRPOS  j, *+2
    SET    j, 55          ; if j = 0, set j <- 55
    BRPOS  k, *+2
    SET    k, 55          ; if k = 0, set k <- 55
\end{verbatim}

\bigskip

The Fibonacci-like generators described above are all ``additive'' in nature. A fundamentally different approach uses \emph{feedback shift register} techniques, which are rooted in the theory of polynomials over finite fields. When $m = 2$, sequences of the form
\[
X_n = (c_1 X_{n-1} + c_2 X_{n-2} + \cdots + c_p X_{n-p}) \bmod 2
\]
can have periods as long as $2^p - 1$, and the theory for choosing the feedback coefficients $c_j$ is well understood. These are called \emph{linear feedback shift register} (LFSR) sequences.

In the context of agentic RL on heterogeneous hardware, the Agent Lightning framework \citep{agentlightning2025} (\texttt{agentlightning/algorithm}, \texttt{agentlightning/runner}) provides a modern instantiation of these generation strategies. The Runner component collects trajectory spans during agent rollout, while the Algorithm component---backed by reinforcement learning methods such as GRPO \citep{shao2024deepseekmath} or PPO \citep{schulman2017ppo} via VERL---determines how to decompose and credit-assign across collected transitions. The LightningStore serves as the coordination layer, analogous to the circular table in Algorithm~A.

\medskip
\noindent\textbf{GRPO: Group Relative Policy Optimization.}\quad Since GRPO is the primary RL algorithm in Agent Lightning's pipeline, we define it precisely. Given a prompt $p$, GRPO generates a group of $G$ candidate responses $\{y_1, \ldots, y_G\} \sim \pi_\theta(\cdot | p)$ and computes a scalar reward $R_i$ for each. The advantage of response $y_i$ is computed \emph{relative to the group statistics}:
\[
\hat{A}_i = \frac{R_i - \bar{R}_G}{\sigma_{R,G} + \epsilon}, \qquad \bar{R}_G = \frac{1}{G}\sum_{j=1}^G R_j, \qquad \sigma_{R,G}^2 = \frac{1}{G}\sum_{j=1}^G (R_j - \bar{R}_G)^2.
\]
The policy gradient is then:
\[
\nabla_\theta \mathcal{L}_{\mathrm{GRPO}} = -\frac{1}{G}\sum_{i=1}^G \hat{A}_i \sum_{t=1}^{T_i} \nabla_\theta \log \pi_\theta(y_{i,t} | p, y_{i,<t}) \cdot \min\!\Bigl(\rho_{i,t}\,,\; \mathrm{clip}(\rho_{i,t}, 1\!-\!\epsilon, 1\!+\!\epsilon)\Bigr)
\]
where $\rho_{i,t} = \pi_\theta(y_{i,t} | \cdot) / \pi_{\theta_{\mathrm{old}}}(y_{i,t} | \cdot)$ is the importance ratio. The critical architectural advantage of GRPO over PPO is the elimination of the learned value function (critic network). Since the critic would consume an entire NeuronCore's \SBUF{} capacity for its own parameter storage and forward pass, GRPO's critic-free design halves the per-device memory requirement during the policy update step---a decisive advantage on Trainium2, where \SBUF{} is the binding memory constraint for large models.

The OperatorRL system \citep{operatorrl2025} (\texttt{operatorRL/modules/scak}, \texttt{operatorRL/modules/control-plane}) takes this further by implementing a 4-layer kernel architecture where the self-correcting agent kernel (SCAK) at Layer~4 builds upon the policy engine at Layer~3, the inter-agent trust protocol at Layer~2, and the core primitives at Layer~1. This layered decomposition mirrors the hierarchical structure of Section~3.2.1's analysis: just as the period length of a composite modulus decomposes via Lemma~Q into the LCM of prime-power periods, the agent system's overall training cycle decomposes into independently optimizable layers.

The critical insight from Anthropic's Trainium2 kernel engineering---as described in the companion document ``What Is Anthropic Really Doing?'' (\texttt{AnthropicWantToDoWhat.txt})---is that in the narrow vertical of AI training and inference, one does \emph{not} need the full CUDA ecosystem (CCCL, cuBLAS, cuDNN, TransformerEngine, NCCL). One needs only to squeeze maximum \MME{} and \GPSIMD{} utilization from a few dozen critical NKI kernels: GEMM (with \SBUF{}/\PSUM{} orchestration), fused attention (CTE and TKG variants for different sequence lengths), and collective communication primitives (over NeuronLink's 800\,GB/s interconnect). Through tight hardware-software co-design with Annapurna Labs, combined with Trainium2's cost-performance advantages and large-scale scalability, one can optimize from the silicon level all the way through the complete software stack.

In Knuth's terms, this is exactly what \emph{The Art of Computer Programming} Vol.~1, \S1.3.1 teaches: understand your machine at the instruction level (the NeuronCore's VLIW pipeline, the DMA engine's multicast and scatter-gather capabilities, the \MME{}'s systolic array scheduling), then perform full-stack optimization for your specific workload. Anthropic does not need a general-purpose GPU programming platform; they need only to make Claude's training and inference run faster than anyone else's, and NKI assembly on Trainium2 is their chosen instrument.

The multi-platform strategy (GPU/TPU/Trainium) simultaneously maintained by Anthropic reveals a genuine engineering pain point: you maintain three bottom-level stacks, each requiring assembly-level experts (CUDA PTX, XLA HLO, Trainium2 VLIW ISA), and the debug space is a Cartesian product---$O(3 + k + d)$ in the best case (when failure modes are independent across platforms) and $O(3 \cdot k \cdot d)$ in the worst case (when cross-platform interaction bugs create combinatorial explosion). This is extraordinarily expensive in kernel engineering terms. But Anthropic is willing to pay this cost, because the strategic bet is on never being locked into any single hardware vendor.

To summarize in one sentence: DeepSeek optimizes brilliantly within the NVIDIA ecosystem; Anthropic helps AWS build an alternative compute foundation. Both paths require assembly-level mastery, but the strategic intent is completely different---and the mathematical framework of this chapter (period theory, potency, spectral tests) applies to kernel optimization on \emph{both} platforms.
